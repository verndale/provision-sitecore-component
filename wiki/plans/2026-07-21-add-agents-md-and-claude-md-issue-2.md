## Context

The repo has no agent-facing operating brief. Claude Code / Codex / Cursor currently
reverse-engineer the repo's conventions, commands, and boundaries from `README.md` +
`CONTRIBUTING.md` + the skill. Sibling repos (ai-orchestration, cumulative-conductor)
carry an `AGENTS.md` as the canonical agent guide with `CLAUDE.md` re-exporting it
(`@AGENTS.md`). This change brings that pattern here: a concise brief that **points into**

indexed as `root-doc` nodes.

Outcome: `AGENTS.md` + `CLAUDE.md` exist at root, follow the sibling convention, and the
graph indexes both.

add-only reconcile, push gate, secrets-never-echoed (all in `SKILL.md` / `references/`).

## AGENTS.md — structure (pointer-brief, not duplication)

H1 `# Agent guide` (becomes the graph node label; parallels CONTRIBUTING.md's `# Contributing`).

1. **What this is** — one para: provision Sitecore XM Cloud components from one reviewed
   manifest → CMS (Authoring GraphQL API) **and** a TSX handoff scaffold; one manifest drives
   both sides so the front-end boundary mirrors the CMS by construction. Link → `README.md`.
2. **Layout** — one line each: `src/` (dependency-free CommonJS CLI runtime), `skills/`
   (the skill + `references/` normative docs), `test/` (node:test + golden fixtures), `wiki/`
   (context wiki — start at `wiki/INDEX.md`), `scripts/` (graph/wiki/evals/release
   automation), `evals/` (behavioral skill-eval scenarios pinned in CI).
3. **Run it** — pnpm 10.33.0 via Corepack, Node ≥24.14.0. `pnpm test`, `pnpm commit`,
   `pnpm graph:build` / `graph:view`, `pnpm evals:check`, `bash setup.sh`. Note the CLI is
   runnable with no install: `node src/cli.cjs <plan|check|push> <manifest.json>`
   (`plan` offline, `check` read-only preflight, `push` mutating + gated).
4. **Config / manifest / auth** — config resolution order (one line): `--config` →
   `provision.config.json` → `build.config.json` (requires `stackAdapter: "sitecore-ai"`) →
   `manifest.sitecorePaths`. Manifest is the single reviewed contract → link
   `skills/.../references/manifest-contract.md`. Auth = OAuth2 client-credentials via
   `SITECORE_AUTHORING_*` env in a gitignored `.env` (see `.env.example`, referenced in code
   font not as a link) → link `references/authoring-api.md`.
5. **Hard boundaries** — the load-bearing section:
   - **Add-only reconcile** — never delete/rename/retype CMS items/fields or remove list
     entries (`__Masters`, Allowed Controls, validation bars); conflicts are follow-ups,
     never "fixed." (→ `authoring-api.md`, `SKILL.md`)
   - **Push gate** — never run `push` without the in-session step-6 gate; `check` is the only
     online mode allowed before it. (→ `SKILL.md`)
   - **Secrets never echoed** — never print `SITECORE_AUTHORING_*` values to chat/logs/files.
   - **Don't invent spec values** — Source strings, datasource locations, field types absent
     from the spec are review questions, written verbatim once answered.
   - **Version control (deliver-and-handoff)** — leave an uncommitted working tree + a
     suggested Conventional Commits message; the owner performs commit/push/merge/tag and
     releases (semantic-release on `main`; wiki-sync via PR, never direct push).
   - **Don't hand-edit generated/vendored files** — `wiki/connections*` (regenerated by
     `graph:build`) and `skills/_meta/*` + `retry-contract.md` (vendored from
     ai-orchestration; re-sync from source).
6. **Where to look next** — the six `references/` docs (2 normative: manifest-contract,
   authoring-api), `wiki/INDEX.md` + `node scripts/wiki/navigate.cjs --intent why|wiring|impact`,
   `CONTRIBUTING.md`, `evals/`.

Authoring rule: links should target real graph nodes (`README.md`, `CONTRIBUTING.md`,
`skills/**/*.md`, `wiki/**/*.md`, `src/**/*.cjs`) so they create useful seam edges; mention
non-node files (`.env.example`, `setup.sh`, `package.json`) in code font, not as links
(harmless either way — non-node link targets are silently skipped, never dangling).

## What could break / blast radius

- **Docs-only apart from the one-line `ROOT_DOCS` addition** — no runtime/CLI behavior changes.
- **Graph freshness gate**: `test/graph.test.cjs` asserts committed `graph.json` and
  `wiki/connections*` are byte-identical to a fresh build. So the sequence is: create files →
  edit line 37 → `pnpm graph:build` → commit the regenerated artifacts, or `pnpm test` fails.
- No dangling-edge / coverage-gate risk: `links-to` edges only form to existing nodes
  (build-graph.cjs:202); `root-doc` needs no topic coverage.
