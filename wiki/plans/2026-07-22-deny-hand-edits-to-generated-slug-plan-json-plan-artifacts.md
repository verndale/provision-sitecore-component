# Deny hand-edits to generated `<slug>.plan.json` plan artifacts

## Context

`node src/cli.cjs plan` writes `<slug>.plan.json` next to the manifest
([cli.cjs:205](src/cli.cjs)). Per [SKILL.md:34](skills/provision-sitecore-component/SKILL.md),
that file is part of the review artifact the developer approves at the step-6 gate.
**Corrected rationale (verified this session):** `push` never reads the plan file back —
every mode rebuilds the plan in-memory from the manifest and rewrites the file
(cli.cjs:204-206), so there is *no push bypass*. The rule protects **gate-review artifact
integrity**: an honest-but-drifting agent that "fixes" the plan file instead of the
manifest presents a doctored artifact at the gate. Hand-edits are also futile (overwritten
on every CLI run), so denying them costs nothing legitimate. Joe confirmed proceeding
with this corrected, narrower rationale. Mechanically identical to the just-shipped
Read-tool guard (#20): one `decideFile` rule through already-registered matchers — no new
hook event, matcher, installer change, or re-registration.

## Problem

Agents can hand-edit `<slug>.plan.json` via Edit/Write tools, corrupting the artifact the
step-6 gate approval rests on. Root cause: `decideFile` has no rule for generated plan
artifacts (it covers wiki/graph, vendored, goldens, `.env` only).

## Files

- `scripts/hooks/guard-core.cjs` — new `REASONS.planEdit` + rule in `decideFile`
- `test/hooks.test.cjs` — decision-class tests, both directions (lockstep per AGENTS.md)
- `AGENTS.md` — add `<slug>.plan.json` to the "Don't hand-edit generated or vendored
  files" bullet (Cursor relies on prose alone)
- `skills/provision-sitecore-component/SKILL.md` — one guardrail line (skill travels to
  consumer repos); **risk**: skills-lint may pin byte counts — check `test/` lint
  expectations when editing
- Wiki: journal `2026-07-22-plan-artifact-edit-guard.md`, topic bullet, INDEX line,
  archived plan (`scripts/wiki/archive-plan.cjs`), `pnpm graph:build`
- **No changes**: `pretooluse-guard.cjs`, `install.cjs`, `.claude/settings.json`,
  `.codex/hooks.json`, `decideRead` (plans must stay readable for gate review)

## Assumptions

- `<slug>.plan.json` naming is tool-defined and stable (cli.cjs:205). **Load-bearing.**
- Basename `endsWith(".plan.json")` is precise: goldens (`expected-plan.json`, hyphen)
  and bare `plan.json` (9 chars) don't match; nested paths do.
- The CLI's own writes go through `fs.writeFileSync` under Bash, not Edit tools — the
  guard never blocks legitimate regeneration.
- Scope model parity: deny in tool repo + provisioning repos; plain repos untouched.

## What could break

- False positives on consumer files coincidentally named `*.plan.json` outside
  provisioning repos — avoided by the plain-repo carve-out.
- SKILL.md edit could trip skills-lint byte-count pinning — verify and refresh whatever
  the lint pins, in the same delivery.
- Precondition: the working tree currently holds the uncommitted #20 delivery. **Do not
  start until it's committed (owner's action).** If the tree is still dirty with #20
  changes at execution time, stop and hand back.
